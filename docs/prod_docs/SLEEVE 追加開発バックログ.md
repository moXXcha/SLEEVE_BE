## Epic 優先度 (2025/08/21)

  

以下の基準に基づき、各 Epic の優先度を決定しました。

  

- **依存性:** 他の機能がその Epic に依存しているか。

- **サービスドメインでの機能優先度:** アプリケーションの核となる価値提供、ユーザー獲得・維持、収益化への貢献度。

- **その他基準:** MVP (Minimum Viable Product) の観点、開発の複雑性、リスク。

  

---

  

### 優先度: 最優先 (Highest Priority)

  

- **1.1: ユーザー認証基盤の構築:**

- **理由:** ほとんどの機能の基盤となるため、最優先で構築が必要です。ユーザーがログインできなければ、他の多くの機能は利用できません。

  

### 優先度: 高 (High Priority)

  

- **2.1: コンテンツ発見機能（ログイン不要）:**

- **理由:** ログイン不要で利用できるため、初期のユーザーエンゲージメントを高め、アプリの価値をすぐに提供できます。認証基盤と並行して開発可能です。

- **2.2: アカウント管理機能（ログイン必須）:**

- **理由:** 認証基盤の次にユーザーが自身の情報を管理するために必要となるため、優先度が高いです。

- \*\*2.3: コーデ投稿・管理機能（ログイン必須）:

- **理由:** ユーザー生成コンテンツの核となる機能であり、アプリの魅力を高める上で非常に重要です。認証とアカウント管理に依存します.

  

### 優先度: 中 (Medium Priority)

  

- **3.1: 決済システム連携:**

- **理由:** アイテム出品・取引機能の実現に不可欠な基盤です。出品機能と並行またはその直前に開発を進めるのが効率的です.

- **3.2: アイテム出品・取引機能（ログイン必須）:**

- **理由:** アプリの収益化に直結する重要な機能です。認証、アカウント管理、決済システムに依存します.

  

### 優先度: 低 (Low Priority)

  

- **4.1: 通知・コミュニケーション機能:**

- **理由:** ユーザーエンゲージメントを高める補助的な機能ですが、コア機能が確立された後に開発しても問題ありません。他の機能からのイベント発生に依存します.

- **4.2: 管理者向けモデレーション機能:**

- **理由:** プラットフォームの健全性を保つために重要ですが、ユーザー数やコンテンツ量が増加してから本格的に開発しても遅くありません。初期段階では手動での対応も可能です.
